# AGENTS.md

## 工作原则

在本项目中编写、修改、审查或重构代码时，遵循以下规则。

### 1. 编码前先理解

- 先阅读相关代码、测试、配置和项目文档。
- 不要未经检查就假设项目结构、接口行为或业务规则。
- 优先通过代码库寻找答案。
- 存在多个合理解释且无法从项目确认时，明确说明假设并询问。
- 发现需求与现有实现冲突时，先指出冲突和影响。

### 2. 保持简单

- 使用能够完整满足当前需求的最简单实现。
- 不添加未被要求的功能、配置项或扩展机制。
- 不为一次性逻辑创建不必要的抽象。
- 优先复用项目已有模式、工具和依赖。
- 避免重复代码，但不要为了消除少量重复而过度设计。

### 3. 控制修改范围

- 每一处修改都必须能够追溯到当前任务。
- 不重构、格式化或清理与任务无关的代码。
- 保持现有命名、目录结构、代码风格和架构边界。
- 不删除无法充分理解的代码或注释。
- 可以删除本次修改造成的无用导入、变量和函数。
- 发现无关问题时只需说明，不要擅自修复。

### 4. 定义验收标准

开始复杂任务前，先明确可验证的成功条件。

推荐流程：

1. 确认当前行为和预期行为。
2. 找到最小修改范围。
3. 实现必要改动。
4. 运行相关测试、类型检查和构建。
5. 检查最终 diff，确认没有无关修改。

修复 Bug 时：

- 尽可能先复现问题。
- 优先添加能够证明问题存在的测试。
- 修复后确认该测试通过。
- 运行相关回归测试。

### 5. 测试与验证

- 修改业务逻辑时，补充或更新相关测试。
- 优先运行最相关的测试，再运行更广泛的检查。
- 使用项目已经配置的测试、格式化、Lint、类型检查和构建命令。
- 不为了让测试通过而降低断言、跳过测试或隐藏错误。
- 无法运行验证时，明确说明原因和未验证的风险。
- 不声称未实际运行的测试已经通过。

### 6. 安全修改

- 不覆盖或撤销用户已有的未提交修改。
- 不使用破坏性 Git 命令，除非用户明确要求。
- 不提交密钥、令牌、密码或其他敏感信息。
- 不执行来源不明的脚本或安装不必要的依赖。
- 新增依赖前，确认现有工具无法合理解决问题，并说明原因。
- 涉及数据库迁移、权限、认证、支付或数据删除时，提高验证强度。

### 7. 沟通要求

- 开始工作时简要说明正在检查什么。
- 对复杂任务给出简短、可验证的执行计划。
- 发现风险、冲突或需求缺口时直接指出。
- 不隐藏不确定性，也不要反复询问可从代码库确认的问题。
- 完成后简要说明：
  - 修改了什么；
  - 为什么这样修改；
  - 执行了哪些验证；
  - 是否仍有未验证风险。

## 项目专属规则

本项目的具体要求：

- 技术栈：Next.js 14 App Router、React 18、TypeScript、Tailwind CSS、Framer Motion、Lucide React、DeepSeek API、localStorage。
- 包管理器：npm。优先使用已有 `package-lock.json` 和 `node_modules`，不要无理由更换为 pnpm、yarn 或 bun。
- 开发命令：`npm run dev`，默认本地地址为 `http://localhost:3000`。
- 测试命令：当前未配置独立测试脚本；涉及功能或交互修改时，至少运行构建并进行必要的手动流程验证。
- Lint 命令：`npm run lint`。
- 类型检查命令：`npm run typecheck`。
- 综合检查命令：`npm run check`，会依次运行 lint、typecheck 和 build。
- 构建命令：`npm run build`。代码、类型、API 或资源路径修改后优先运行该命令验证。
- 主要源码目录：`src/app/`、`src/components/`、`src/data/`、`src/lib/`、`src/types/`。
- 项目文档目录：`ProjectDocument/`。修改代码、视觉方案、动效方案、AI 接口或素材策略时，需要同步更新相关文档；阶段进度或下一步变化需要同步更新 `ProjectDocument/HANDOFF.md`。
- 静态资源目录：`public/assets/` 存放背景和卡背，`public/cards/rws/` 存放 78 张 Rider-Waite-Smith 真实牌面。
- 测试目录：当前未设置专用测试目录。
- 禁止修改的目录或文件：不要手动修改 `node_modules/`、`.next/`、构建缓存、临时文件、`.env.local` 或任何包含密钥的文件；不要提交 API Key、令牌、密码或本地私密配置。
- DeepSeek 约束：API Key 只能由服务端读取，前端不得直接调用 DeepSeek 或暴露环境变量；接口失败时保留抽牌状态并允许手动重试，不自动生成模拟解读。
- 视觉约束：整体保持“旧女巫木桌风”，避免现代霓虹、科技星空、大面积玻璃拟态、高饱和渐变和全屏粒子；真实牌面需要保持旧牌外框、暗金描边和复古滤镜。
- 交互约束：优先保证完整占卜主流程可跑通，包括提问、洗牌、抽牌、翻牌、AI 解读、命运手记和历史保存。
- Git 规则：所有代码或项目文档修改完成并验证后，需要提交 Git 版本并推送到 GitHub；提交前检查范围，避免纳入无关文件。

## 指令优先级

- 用户在当前任务中的明确要求优先。
- 更接近目标文件的子目录 `AGENTS.md` 可以提供更具体的规则。
- 若规则发生冲突，应指出冲突，并采用范围更具体且不违反用户要求的规则。
